Why HTML beats Markdown for an LLM's memory

An AI agent accumulates code knowledge and conversation knowledge every day. Markdown is the default substrate for storing it. It shouldn’t be. A data-backed case with real numbers.

1. The setup: what the memory problem actually is

An AI coding agent is not stateless. Every session produces code edits, architectural decisions, bug discoveries, and chosen trade-offs. The question is not whether to remember them — it is how.

The naive answer is: Markdown notes, like every developer note system since 2004. One file per topic, YAML frontmatter for metadata, keyword search to retrieve. This is what Obsidian, basic-memory, and most “second brain” tools for LLMs do.

The problem is that LLM memory is different from human memory in one critical dimension: every token you retrieve costs money and dilutes the context window. A human reading a 2,000-word note skips irrelevant paragraphs automatically. An LLM receiving a 2,000-word note processes every token at the same cost.

This means the substrate you choose for memory has a direct, measurable impact on:

  • API cost — every unnecessary token in context is billed
  • Answer quality — irrelevant tokens dilute signal
  • Query precision — returning the right thing, not a superset
  • Temporal correctness — knowing when a fact was true, not just that it exists

Markdown addresses none of these constraints structurally. HTML does — if you use it correctly. The rest of this document explains exactly how, with numbers.

2. The tag is the index

In LazyBrain, every fact is an HTML element — not a Markdown bullet, not a JSON record, not a graph node. An HTML element annotated with data-cerveau-* attributes that declare what the element is, when it was true, who extracted it, and how confident we are.

Here is a real neuron from the sample brain:

<article id="oauth-migration-12"
         data-cerveau-type="decision"
         data-cerveau-created="2026-05-20T10:15:00Z"
         data-cerveau-valid-from="2026-05-20"
         data-cerveau-tags="auth oauth security migration"
         data-cerveau-importance="0.9">

  <h2>Migration to OAuth2 PKCE</h2>

  <p data-cerveau-fact data-cerveau-confidence="1.0"
     data-cerveau-extracted-by="human"
     data-cerveau-kind="decision">
    Decision: drop custom JWT in favour of OAuth2 PKCE.
  </p>

  <p data-cerveau-fact data-cerveau-confidence="0.8"
     data-cerveau-extracted-by="heuristic">
    Driver: Q2 security audit flagged manual secret rotation.
  </p>

</article>

When the LLM needs this fact, the system does not send the HTML. It runs a CSS selector and extracts only the textContent: 

// Query: all still-valid auth decisions
document.querySelectorAll(
  'article[data-cerveau-type="decision"]' +
  '[data-cerveau-tags~="auth"]' +
  ':not([data-cerveau-valid-until])'
)

// What the LLM receives (stripped):
[decision | 2026-05-20 | oauth-migration-12]
Migration to OAuth2 PKCE
Decision: drop custom JWT in favour of OAuth2 PKCE.
Driver: Q2 security audit flagged manual secret rotation.

The markup is the index. The inner text is the payload. Every data-cerveau-* attribute is a zero-cost, instantly-queryable column that costs nothing extra to store and nothing to query.

The key insight: HTML gives you a structured index inside the document format itself. You do not need a separate database, a vector index, or a metadata sidecar file. The format and the index are the same artifact.

Markdown has no equivalent. A Markdown file can have YAML frontmatter, but frontmatter is a convention with no standard parser and no native query semantics. You cannot run article[type="decision"]:not([valid-until]) on a folder of Markdown files. You can only keyword-grep.

4. Exact, typed, temporal precision

Some queries are not fuzzy. They have exact, unambiguous answers:

  • “Show me all decisions that are still active (not superseded).”
  • “What was the state of the auth module as of 2026-03-01?”
  • “Find every bug logged in the Acme project.”

Markdown cannot answer these precisely. Keyword-grep on “decision” returns every file that mentions the word — including comments, changelog entries, meeting notes where decisions are discussed but not made. There is no structural distinction between “a file that IS a decision” and “a file that mentions the word decision.”

HTML with data-cerveau-* attributes answers them with zero false positives:

/* All still-valid decisions — native CSS, works in any browser */
article[data-cerveau-type="decision"]:not([data-cerveau-valid-until])

/* All bugs in the Acme project — native CSS, works in any browser */
article[data-cerveau-tags~="bug"][data-cerveau-tags~="acme"]

/* Auth facts that became valid in 2026 and are not yet expired
   — range predicate, evaluated in JS over attributes
   (native CSS stops at exact match; LazyBrain's query engine adds ranges) */
Array.from(document.querySelectorAll(
  'article[data-cerveau-tags~="auth"]:not([data-cerveau-valid-until])'
)).filter(el =>
  (el.getAttribute('data-cerveau-valid-from') || '') >= '2026-01-01'
)

/* Time-travel: reconstruct brain state as of 2026-03-01
   — range predicate, evaluated in JS over attributes */
Array.from(document.querySelectorAll('article[data-cerveau-valid-from]'))
  .filter(el => {
    var from  = el.getAttribute('data-cerveau-valid-from')  || '';
    var until = el.getAttribute('data-cerveau-valid-until') || '';
    return from <= '2026-03-01' && (until === '' || until >= '2026-03-01');
  })

A CSS predicate cannot return a non-matching element by construction. Precision is 100% for these structural/typed queries — not as an empirical claim, but as a logical property of set membership. The benchmark’s structural queries (SQ1–SQ4) demonstrate this on real data:

Query HTML predicate What Markdown keyword-grep gets wrong
Active concepts (not expired) :not([data-cerveau-valid-until]) Returns any file containing “concept” — structure pages, code comments, README text
Bugs in Acme [tags~="bug"][topic~="acme"] Matches “bug” in unrelated error messages, code comments, any doc mentioning bugs
Decisions since May 2026 .filter(el => el.getAttribute('data-cerveau-created') >= '2026-05-01')
range predicate in JS — native CSS has no ≥/≤ operator		 “2026-05” appears in timestamps, URLs, and file paths of any document
Modules in content-pipeline [type="aggregate-neuron"][code-path~="content-pipeline"] Returns any doc mentioning “content-pipeline” regardless of node type

Time-travel is a first-class CSS selector. Competing LLM memory systems that extract facts from prose struggle with temporal reasoning: mem0’s LoCoMo temporal sub-score is 55.51%; Letta/MemGPT temporal F1 is 25.52 — both systems’ weakest dimension. In LazyBrain, valid-from/valid-until are attributes. The answer is either in the attribute predicate or it isn’t.

Scope: The 100% precision claim applies strictly to structural and typed queries — queries expressible as attribute predicates. Free-text questions (“when did we decide to change the caching strategy and why?”) still require keyword (L2), semantic (L3), or LLM (L5) retrieval, where precision is not guaranteed. LazyBrain has not been evaluated on LoCoMo, LongMemEval, or DMR. No claims about free-text accuracy relative to competitors are made here.

5. Deterministic and free to query

When you send the same query to mem0, cognee, graphiti, or Letta twice, you may get different results. LLM inference is stochastic; extraction and re-ranking introduce variance. This is not a criticism — it is a design property of systems that use LLMs as query processors.

LazyBrain’s L1 (CSS selector) and L2 (SQLite FTS5) layers are deterministic by construction. The same brain state + the same query = the same bytes in the same order, every time. scripts/bench-determinism.mjs verifies this:

  SHA-256 : 6d2fb562992ead24b1c9413d4c651d528107f0fef481931b2743fa93ce3fc300

Determinism matters for three practical reasons:

  • CI regression testing — a changed brain changes the hash; a CI job detects unintended mutations without re-running an LLM judge.
  • Safe caching — query(brain_hash, selector) → result_hash is a pure function; any cache can store the mapping forever without staleness risk.
  • Auditability — a stored hash proves the brain state at a point in time; you can reconstruct the exact context that informed any decision.

6. A brain you can read, share, and host

HTML is not just machine-readable — it is human-readable in the most literal sense: any browser can render it. This is not a small thing. It means the same file that gets queried by lazybrain search --strip is the page you browse in the wiki. No separate “human interface.” The storage format IS the interface.

No competing system shares this property:

  • Vector DBs (Qdrant, Pinecone, Weaviate): opaque binary indexes. Cannot be opened in a browser.
  • Graph DBs (Neo4j, FalkorDB): require running servers and specialized query clients.
  • LLM memory layers (mem0, cognee, Letta): memory blocks are not human-readable without the app.
  • Markdown tools (Obsidian, Logseq): human-readable, but not LLM-efficient without a separate pipeline.

Sharing is trivial — the brain is plain files on disk: 

brain/
  auth/
    oauth-migration-decision.html
    jwt-removal-procedure.html
  acme/
    architecture-overview.html
    stripe-integration.html

# Share it
zip -r my-brain.zip brain/

# Put it in git (diffable, reviewable)
git add brain/ && git commit -m "chore: update brain" && git push

# Publish it as a website
lazybrain publish --output ./public
# Deploy to GitHub Pages, Netlify, Cloudflare Pages, Vercel, S3

The recipient can read any .html file in a browser, run lazybrain search against it, or diff specific neurons with standard git tools. No installation beyond Node.js.

Long-term durability: HTML and CSS are standardized by the W3C with a 30-year backwards-compatibility track record. A brain from 2026 will be fully readable in a browser in 2046. data-cerveau-* attributes are standard HTML5 data-* attributes — any HTML parser that exists will preserve them. A Qdrant vector index from 2026 is not guaranteed to be readable by anything in 2046.

LazyBrain is the only system where “human reading the brain” and “LLM querying the brain” are the same operation on the same file.

8. Honest closing: what HTML does not solve

This thesis would be dishonest without stating clearly what HTML and CSS selectors cannot do.

Fuzzy semantic synthesis

“Summarize everything I’ve learned about auth across all projects” cannot be answered by a CSS selector. It requires keyword search (L2) or embedding-based semantic search (L3), followed by an LLM synthesis pass. HTML’s structured attributes help retrieve the right facts, but the synthesis is still an LLM operation.

Untagged content

The data-cerveau-* advantages only apply to content that has been processed by the dream and enrich pipelines. Raw notes that have not been tagged fall back to keyword search only. The format is only as good as the tagging discipline.

Free-text temporal reasoning

“When did the team decide to stop using JWT?” — if the decision is stored as a tagged fact with data-cerveau-valid-until, a CSS selector answers it exactly. If the decision is buried in prose without attributes, the same limitations as any full-text search apply.

No public benchmark numbers yet

LazyBrain has not been evaluated on LoCoMo, LongMemEval, or DMR. All competitor accuracy numbers cited here belong to the competitors. The structural precision claims (100% on typed queries) are logical properties, not benchmark results. A LoCoMo evaluation is planned but not yet done.

The claim, honestly stated: for the specific capabilities it provides — typed queries, temporal predicates, determinism, token efficiency, portability, browsability, and durability — HTML is the right choice, and no alternative format provides all of them simultaneously. It is not a claim that HTML solves everything.
